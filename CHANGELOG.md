# 변경 이력

## v0.5 — 보안/정합성 강화
### 보안
- **`clientId` 임퍼스네이션 제거**: 브라우저가 자칭하던 `clientId`를 신뢰하지 않는다. 소켓
  연결마다 서버가 HMAC 서명 세션 토큰을 발급·검증해 신원을 결정하고, 모든 뮤테이션
  (`party:create/join/leave/settings`)은 페이로드의 `clientId`를 완전히 무시한 채 그 신원만
  사용한다. 재접속(새로고침 등)에는 토큰 재전송으로 신원이 유지된다.
- **방 DTO에서 신원 정보 제거**: `MemberDTO`는 더 이상 어떤 식별자도 담지 않는다. 보는 사람마다
  개인화된 `mine`(내 항목인지)·`isOwner`(내가 방장인지) 플래그만 내려준다. 방 상태 브로드캐스트에는
  세션 자격증명(토큰)이 절대 실리지 않는다(별도 1:1 `session:token` 이벤트로만 전달).
- **방장 전용 설정**: `Party.ownerId`(만든 사람) 추가. `party:settings`는 방장만 허용하고
  아니면 `NOT_OWNER`. 방장이 나가면 최선임 멤버에게 자동 위임.
- **Socket.IO 룸 누수 차단**: 소켓이 다른 방으로 갈아탈 때 이전 룸에서 명시적으로 `leave()`한다.
- **소켓당 레이트 리밋**(고정 윈도, 기본 20회/10초, `RATE_LIMIT_MAX`/`RATE_LIMIT_WINDOW_MS`)과
  **Origin 화이트리스트**(`allowRequest`로 polling·WebSocket 모두 커버, `ALLOWED_ORIGINS`
  미설정 시 안전한 기본값인 same-Host만 허용, **Origin 헤더가 없는 요청은 항상 거부** — 실제
  브라우저는 두 전송 모두에 Origin을 반드시 실으므로 우회 수단이 되지 않는다)를 추가.
- **[리뷰 발견/수정] SESSION_SECRET 빈 문자열 처리**: `.env.example`이 안내하는 대로
  `SESSION_SECRET=`을 빈 값으로 두면 `??` 연산자 특성상 "미설정"이 아니라 빈 문자열 그 자체가
  HMAC 시크릿으로 쓰여, 위조 가능한 예측 가능한 키가 되는 문제가 있었다. `||`로 교체해 빈
  문자열도 미설정과 동일하게 취급하도록 수정, 위조 회귀 테스트로 검증.
- **[리뷰 발견/수정] Origin 부재 우회 구멍**: 최초 구현은 Origin 헤더가 없는 요청을 무조건
  허용해, `ALLOWED_ORIGINS`를 설정해도 Origin만 생략하면 화이트리스트를 통째로 우회할 수
  있었다. 비브라우저 클라이언트를 위한 예외를 별도로 두지 않고 Origin 부재를 항상 거부하도록
  수정(정말 필요하면 그 클라이언트가 Origin을 직접 보내 화이트리스트에 등록하는 편이 항상 더
  안전하다는 판단, YAGNI). polling·WebSocket 두 전송 모두에 대한 회귀 테스트 추가.
### 수정
- **칼바람 → 라인 큐 전환 버그**: 라인을 고른 적 없는 멤버가 라인 있는 큐로 전환된 뒤에도
  정원만 차지하고 어떤 라인에도 안 보이던 문제. 전환 시 라인 없는 멤버를 파티에서 제거한다.
### 내부
- CI가 `npm run test:cov`(라인/함수/브랜치/구문 80% 임계, `vitest.config.ts`)와
  `npm audit --omit=dev --audit-level=high`를 강제하도록 `.github/workflows/ci.yml` 변경.
### 검증
- 도메인/서버/통합 테스트 **106개** 통과, `tsc` clean, 전체 커버리지 라인 98%·브랜치 92%·
  함수 97%·구문 98%(모두 80% 임계 상회), `npm audit --omit=dev` 0건.

## v0.4.1 — 라인 재선택 버그 수정 + 문구/크레딧
### 수정
- **라인 재선택 버그**: 신청 현황에서 `×`로 라인을 뺀 뒤, 다시 위 토글로 그 라인을 선택하고 "내 라인 저장"을 눌러도 추가되지 않던 문제 해결. 원인은 `detailSel`을 새 `Set`으로 재할당하면서 토글 클릭 리스너가 옛 `Set`을 참조하던 것 — 재할당 없이 제자리 갱신하고 리스너를 1회만 연결하도록 수정.
### 변경
- 제목 **롤챠쿠챠**, 닉네임 입력 안내문구 **카카오톡 닉네임을 입력**, 목록 제목 **파티 목록**, 하단 **만든사람: hong, SED** 표기.
- 시간 입력 옆 **닫기** 버튼 추가(달력 닫기).

## v0.4 — 라인 다중 선택 참가 + 시간 필수
### 변경
- **참가 방식 = 라인 다중 선택**: 탑·정글·미드·원딜·서폿·ALL 토글로 갈 라인을 여러 개 고르고 **참가/내 라인 저장** 버튼으로 신청. 한 사람이 여러 라인 가능.
- **신청 현황을 라인별 이름 칩으로 표시**: 내 칩은 `×`로 해당 라인에서 빼기(마지막 라인을 빼면 파티에서 나감). 서로 다른 사람이 같은 라인 공유 가능.
- **방 만들 때 시간 필수**(input `required` + 검증). 상세에서도 시간 비우기 불가.
- 5명 인원 제한 유지(사람=`clientId` 기준 5명, 재참가 시 닉네임·라인 통째 갱신).
- 칼바람(ARAM)은 라인 선택 숨김 + 인원 명단만.
### 내부
- 도메인 `Member.position`(단일) → `positions`(배열, 중복 제거·표준 순서 정렬), 이벤트 payload `position` → `positions`.
### 검증
- 도메인/서버 테스트 28개 통과, `tsc` clean, Playwright 실측(다중 라인 생성·공유 라인·`×` 드롭·ARAM 명단·시간 필수·오버플로 0).

## v0.3.1 — 방 기억하기
### 수정
- **맨 URL 접속 시 방이 매번 새로 생기던 문제**: 이제 브라우저가 마지막에 쓰던 방(`room` 코드)을 기억해, 도메인만 입력해도 그 방으로 복귀한다. 만든 파티가 사라지지 않음. (친구는 여전히 **`링크 복사`** 한 링크(`?room=코드`)로 들어와야 같은 목록을 봄. 새 방은 **`새 방`** 버튼.)

## v0.3 — 파티 목록판 + 다중 참가 + 반응형
### 추가
- **파티 목록판**: 한 링크가 곧 하나의 게시판. 하루에도 여러 파티를 만들 수 있고, 카드를 눌러 **상세**로 들어가 자리를 채운다. **뒤로가기(← 목록/폰 뒤로가기)** 로 목록 복귀(History API).
- **같은 포지션 다중 참가**: 한 포지션에 여러 명 가능, 단 **파티 총원 5명** 상한. `clientId` 1인 1자리(다른 포지션 누르면 이동).
- **칼바람(ARAM) 인원 명단 모드**: 큐가 칼바람이면 포지션 없이 인원(최대 5명)만. 생성 폼에서 포지션 필드 자동 숨김, 큐를 칼바람으로 바꾸면 기존 포지션도 비움.
- 도메인 `party`(파티·멤버·정원·포지션 유무), 오류 코드 `PARTY_FULL/PARTY_NOT_FOUND`.
- 이벤트 교체: `party:create` · `party:join` · `party:leave` · `party:settings`, 상태 브로드캐스트 `room:state`.

### 변경
- **모델 전환**: `방 = 단일 게임` → `방 = 파티 리스트`. 만료(예정 시각 경과) 파티는 목록에서 자동 제거, 빈 방은 메모리 정리.
- **날짜 표기에서 연도 제거**: 앱 표시(카드·마감)를 `월/일 + 오전·오후 시:분`(예: `7/9 오후 8:30`)으로. (기기 기본 날짜 선택 위젯 자체 표기는 브라우저 영역.)
- **카카오톡 인앱 브라우저 대응(반응형)**: body를 flex 센터링으로 바꿔 grid auto-track 오버플로 제거, `min-width` 체인·`minmax(0,1fr)`, 입력 폰트 16px(iOS 확대 방지), `safe-area`/`100dvh` 폴백. **320px에서 가로 오버플로 0** 확인.
- 닉네임 클라이언트측 HTML 이스케이프(XSS 방지) 추가.

### 검증
- 도메인/서버 TDD로 재작성, **27개 테스트 통과**, `tsc` clean.
- Playwright 실측(375·320px): 목록↔상세·뒤로가기, 소켓 경유 중복 포지션·5명 cap(6번째 `PARTY_FULL`), 칼바람 명단, 날짜 표기, 오버플로 0.

### 알려진 한계
- 파티 설정 수정은 현재 방 참여자 누구나 가능(친구용 오픈채팅 전제). 소유자 제한은 후속 과제.

## v0.2 — 저장형 모집 + 설정 + 게임 수명
### 추가
- **티어 선택**: 상관없음·아이언·브론즈·실버·골드·플래티넘·에메랄드·다이아몬드·마스터·그랜드마스터·챌린저.
- **큐 선택**: 일반게임·칼바람·솔로랭크·자유랭크·5인 자유랭·기타.
- **시간 설정 작동**: 예정 시각을 정하고, 그 시각까지 자유 수정. 마감까지 남은 시간 카운트다운.
- **게임 수명**: 예정 시각이 지나면 현재 게임이 마감되고 **새 게임(빈 자리)**이 자동으로 열림(헤더 "N번째" 증가).
- **킵얼라이브**: GitHub Actions 워크플로 + UptimeRobot 안내(docs/DEPLOY.md).
- 도메인: `tiers` · `queues` · `game`(설정·수명) 모듈, 오류 코드 `INVALID_TIER/QUEUE/TIME`.
- 이벤트 `settings:update` 추가.

### 변경
- **실시간 → 저장형**: 자리는 연결과 무관하게 **저장**되어 창을 닫거나 나중에 들어와도 유지(실시간 반영은 유지).
- **소유권을 소켓 id → 브라우저 `clientId`(localStorage)** 로: 새로고침·재접속해도 본인 자리 유지·해제 가능.
- 연결 종료로 자리를 풀지 않음(아무도 안 쓴 기본 방만 정리).
- 배포 브랜치를 `main`으로, `render.yaml` 반영.

### 검증
- 도메인 TDD 13개 추가, 총 **52개 테스트 통과**, `tsc` clean, 브라우저 검증.
